# Skill: PR Review

Review a pull request in the **cyfor-ai-workshop** repository.
Produce a concise review that highlights real issues — bugs, correctness
problems, security risks, missing steps — rather than style nits.

## How to invoke

When the user asks to review a PR, fetch the diff and changed file list
from GitHub, then evaluate the changes against every section below.

## 1. Code-generation pipeline

This project drives the frontend from an OpenAPI spec generated by the API.
The chain is: Zod schemas → `createRoute` → `openapi.json` → Orval hooks.

- If the PR touches **any** route definition or Zod schema in
  `api/src/app.ts`, the generated OpenAPI doc (`api/openapi.json`) and
  Orval hooks (`web/src/api/generated/`) **must** be regenerated
  (`npm run generate`). Flag the PR if regenerated files are missing.
- Files under `web/src/api/generated/` must **never** be hand-edited.
  If the diff shows manual changes there, reject immediately.
- If a route is renamed or removed, verify that all frontend consumers
  of the old generated hook have been updated.

## 2. API conventions

- Every public endpoint uses `createRoute(...)` registered via
  `app.openapi(route, handler)`. Plain `app.get/post/…` bypasses the
  OpenAPI doc and breaks codegen — flag it.
- Zod request/response schemas include `.openapi('Name')` so Orval
  emits named TypeScript types.
- Prisma `Date` fields are converted to ISO strings before responding
  (see `toResourceResponse`). Response schemas use
  `z.string().datetime()`. New models must follow the same pattern.
- Path parameters use `z.coerce.number()` because Hono delivers them
  as strings.
- Relative imports **must** include the `.js` extension (`./db.js`, not
  `./db`). This is required by NodeNext ESM resolution.

## 3. Web / frontend conventions

- Components consume data through generated hooks re-exported from
  `web/src/api/index.ts`. Direct `axios` calls in components are not
  allowed.
- After a mutation, invalidate the relevant query using the generated
  query-key helper (e.g. `getGetResourcesQueryKey()`).
- Tailwind v4 is configured through `web/src/index.css` — there is no
  `tailwind.config.js`. If the PR adds one, flag it.

## 4. Database & schema

- The project uses **`prisma db push`**, not migrations. If the PR adds
  a migration file, flag it.
- `api/openapi.json` is generated — hand-edits will be overwritten.
- Schema changes in `api/prisma/schema.prisma` should match the Zod
  schemas in `api/src/app.ts` (field names, types, nullability).

## 5. Domain validation (booking / resource management)

- Field constraints should be sensible for the domain:
  - `title`: required, 1–120 chars
  - `description`: optional/nullable, max 500 chars
  - `category`: required, 1–60 chars
- New entities should follow the existing `Resource` pattern unless
  there is a clear reason to diverge.
- Check that create vs. update schemas have correct required/optional
  semantics (create requires mandatory fields; update makes them
  optional for partial updates).

## 6. General quality

- Changes should be **small, focused, and easy to review**. If a PR
  mixes unrelated concerns, note it.
- Look for obvious bugs, missing error handling, unvalidated inputs,
  and risky assumptions (e.g. assuming a record exists without
  checking).
- No secrets, credentials, or tokens should appear in the diff.
- Edge cases to watch: empty strings that pass `.min(1)` after
  trimming, integer overflow on IDs, concurrent delete + update races.

## Output format

Structure the review as:

1. **Summary** — one-paragraph assessment (approve / request changes /
   comment only).
2. **Issues** — numbered list of problems, each with the file, line
   range, and a concrete suggestion. Omit this section if there are no
   issues.
3. **Nits** — minor observations that don't block merging. Keep this
   short or omit entirely.
